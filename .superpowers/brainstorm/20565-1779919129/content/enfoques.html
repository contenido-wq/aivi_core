3 Enfoques de Arquitectura

¿Cómo construimos el sistema de trazabilidad?

A

Enriquecer tabla existente (más simple)

Agregar columnas nuevas directo a subscriptions: phone, country, city, channel, industry, objective, admin_notes. El webhook de Hotmart las llena automáticamente.

Pros

  • Implementación rápida
  • Sin migración compleja
  • Una sola tabla a consultar

Contras

  • Mezcla datos de pago con perfil
  • Difícil escalar a futuro
  • Sin historial de cambios
B

Tabla user_profiles dedicada ⭐ (Recomendado)

Nueva tabla user_profiles vinculada por email. Separación limpia: transactions = pagos, subscriptions = estado, user_profiles = quién es esta persona. Admin CRM completo con UI dedicada dentro del AdminPanel.

user_profiles
email → buyer_phone, country, city, platform_channel
profile_industry, profile_objective, admin_notes
first_purchase_at, months_active, ltv_usd
usage_tools (JSON), support_history (JSON)

Pros

  • Arquitectura limpia y escalable
  • LTV calculado automáticamente
  • Admin puede editar sin tocar pagos
  • Fácil agregar más campos

Contras

  • Una migración más
  • Lógica de sync entre tablas
C

CRM completo con timeline de eventos (más potente)

Además del perfil, una línea de tiempo cronológica de todo lo que ha hecho el usuario: primera compra, renovaciones, features usadas, notas del admin. Como un mini-CRM interno.

Pros

  • Visión 360° del cliente
  • Ideal para retención
  • Detecta usuarios en riesgo

Contras

  • Mayor complejidad
  • Más tiempo de desarrollo
  • Puede ser overkill ahora

💡 Mi recomendación: Opción B — arquitectura sólida sin sobre-ingeniería. La Opción C puede agregarse encima después.
